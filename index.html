Azure Landing Zones

En god Azure-plattform starter med riktig grunnmur. Landing Zones gir struktur før kompleksiteten tar styringen.

Plattform før prosjektkaos

Jeg bruker Azure Landing Zones til å etablere tydelige rammer for abonnementer, nettverk, identitet, sikkerhet, logging, policyer og ansvarsdeling.

Det gir virksomheten en plattform som applikasjonsteam kan bygge på, uten at hvert prosjekt må finne opp sikkerhet, governance og drift på nytt.

Eksempel: Management groups, hub-spoke nettverk, standardiserte abonnementer, policyer og logging gir bedre kontroll og mindre behov for brannslukking senere.

Terraform og IaC

Infrastruktur som kode gjør Azure-miljøer mer repeterbare, sporbare og kontrollerbare.

Fra manuelle klikk til kontrollerte endringer

Terraform gir mulighet til å bygge og endre infrastruktur på en forutsigbar måte. Det reduserer manuelle feil og gjør det enklere å forstå hva som faktisk er deployet.

Jeg legger vekt på struktur, moduler, navnestandarder og tydelig eierskap, slik at IaC ikke bare blir kode, men et styringsverktøy.

Eksempel: Resource groups, nettverk, Key Vault, policyer og monitorering kan etableres likt hver gang, med kontroll før endringer rulles ut.

Automatisering

Automatisering handler ikke bare om fart. Det handler om færre feil, mindre personavhengighet og bedre drift.

Stabil drift krever repeterbare prosesser

Jeg automatiserer gjentakende oppgaver for å redusere manuell håndtering og gjøre miljøene mer robuste.

Når oppgaver blir standardisert og dokumentert i kode eller scripts, blir drift mindre sårbar og enklere å overlevere.

Eksempel: Automatisering av brukerhåndtering, servervedlikehold, konfigurasjon og rapportering frigjør tid og gir bedre kvalitet.

Azure DevOps

Gode pipelines gir tryggere leveranser, bedre kvalitet og full sporbarhet fra kode til produksjon.

Kontrollert leveranseflyt

Jeg bruker Azure DevOps til å skape struktur i leveranser, fra backlog og kode til bygg, test og deploy.

Pipelines gir kvalitetssikring før endringer slippes videre, særlig når infrastruktur og sikkerhet også leveres som kode.

Eksempel: Terraform-validering, plan, godkjenning og deploy i pipeline gir bedre kontroll enn manuelle endringer direkte i portalen.

Containerteknologi

Containerteknologi gir moderne måter å pakke, deploye og drifte applikasjoner på.

Docker, Kubernetes og praktisk forståelse

Jeg har grunnleggende erfaring med Docker og Kubernetes, og ser containerteknologi som et naturlig supplement til Azure, DevOps og automatisering.

Jeg har brukt kubectl til å inspisere ressurser og forstå hvordan applikasjoner kjører i Kubernetes-miljøer.

Eksempel: Docker kan pakke applikasjoner med avhengigheter, mens Kubernetes håndterer deployering, skalering og drift av containere.

Governance

Governance er forskjellen på en skyplattform som kan styres, og en skyplattform som bare vokser.

Kontroll uten å stoppe fremdrift

Jeg jobber med governance som gjør det mulig å bygge raskt, men innenfor trygge rammer. Det handler om policyer, roller, standarder, kostnadskontroll og tydelig ansvar.

God governance skal ikke være en brems. Den skal være autovernet som gjør at farten kan økes uten at alt havner i grøfta.

Eksempel: Policyer kan sikre tagging, begrense regioner, hindre feilkonfigurasjon og gi bedre oversikt over kostnader og eierskap.

Sikkerhet

Sikkerhet må bygges inn i plattformen, ikke legges på som pynt etterpå.

Zero Trust og innebygd sikkerhet

Jeg jobber med Zero Trust, minst mulige privilegier, segmentering, logging, monitorering og sikker håndtering av hemmeligheter.

Sikkerhet må være praktisk nok til at den faktisk blir brukt, og tydelig nok til at den kan etterleves.

Eksempel: Entra ID, Conditional Access, Key Vault, logging og policyer gir bedre kontroll på tilgang, hemmeligheter og hendelser.

IAM

Identity and Access Management er en av de viktigste kontrollflatene i Azure. Feil tilgang er ofte større risiko enn feil teknologi.

Riktig tilgang, riktig nivå, riktig tidspunkt

Jeg jobber med IAM som en del av plattformarkitekturen, ikke som en isolert administrasjonsoppgave. Tilgang må styres, begrunnes, logges og kunne fjernes.

I Azure betyr dette god bruk av Entra ID, RBAC, Privileged Identity Management, Conditional Access, managed identities, service principals og tydelig rollemodell.

Målet er å redusere permanent tilgang, begrense risiko og gjøre det enklere å revidere hvem som har tilgang til hva.

Eksempel: Utviklere trenger ikke permanent produksjonstilgang. Med PIM kan tilgang gis midlertidig, med godkjenning, logging og automatisk utløp.

Hybrid drift

Mange virksomheter må leve godt med både eksisterende infrastruktur og moderne skyplattformer.

Bro mellom gammelt og nytt

Hybrid drift handler om å få identitet, nettverk, sikkerhet og drift til å fungere på tvers av lokale miljøer og Azure.

Jeg har erfaring med Microsoft-miljøer, Windows Server, AD DS, Entra ID og overgangen fra tradisjonell drift til mer skybaserte plattformer.

Eksempel: Integrasjon mellom on-prem Active Directory og Entra ID kan gi brukerne en mer sømløs hverdag, samtidig som virksomheten moderniserer i riktig tempo.

CAF

Cloud Adoption Framework gir struktur for skyreisen, fra strategi til drift.

Skyadopsjon med retning

Jeg bruker CAF som et praktisk rammeverk for å koble tekniske valg til virksomhetens mål, organisering og modenhet.

Det gir en helhetlig tilnærming til strategi, plattform, governance, sikkerhet, migrering og drift.

Eksempel: CAF kan brukes til å planlegge landing zones, roller, driftsmodell, governance og prioritering av workloads.

WAF

Well-Architected Framework hjelper med å finne svakheter før de blir dyre problemer.

Arkitektur som tåler drift

Jeg bruker Well-Architected Framework til å vurdere løsninger opp mot sikkerhet, pålitelighet, kostnadsstyring, operasjonell kvalitet og ytelse.

Det gir et godt grunnlag for å forbedre løsninger før små tekniske valg blir store driftsutfordringer.

Eksempel: En løsning kan vurderes for redundans, monitorering, kostnadskontroll, sikkerhet og evne til å håndteres i produksjon.
