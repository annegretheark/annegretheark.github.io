Azure Landing Zones

Jeg bruker Azure Landing Zones til å etablere struktur som gir kontroll på sikkerhet, kostnader, drift og ansvarsdeling.

Struktur som fungerer i praksis

Azure Landing Zones gir en gjennomtenkt struktur for abonnementer, nettverk, identitet, sikkerhet, policyer og styring. Det gjør det enklere å bygge videre uten at plattformen blir uoversiktlig.

Jeg jobber med å etablere plattformsoner som gir tydelig ansvarsdeling mellom plattformteam og applikasjonsteam, samtidig som governance og sikkerhet er bygget inn fra starten.

Eksempel: Et oppsett med management groups, policyer, hub-spoke nettverk og standardiserte abonnementer for produksjon, test og utvikling gir bedre kontroll og enklere forvaltning over tid.

Terraform og IaC

Infrastruktur som kode gir repeterbarhet, sporbarhet og færre manuelle feil.

Automatiser infrastrukturen

Jeg bruker Terraform til å standardisere og automatisere Azure-miljøer på en konsistent og kontrollert måte. Det gjør det enklere å redusere feil, jobbe mer effektivt og få bedre kontroll på endringer.

Med gjenbrukbare moduler og tydelig struktur kan nye miljøer etableres raskt og likt hver gang.

Eksempel: Opprette resource groups, nettverk, Key Vault, monitorering og policy-tilknytning automatisk gjennom kode, slik at etablering og videre drift blir mer forutsigbar.

Automatisering

Automatisering er viktig for å redusere feil, øke effektivitet og sikre stabil drift.

Effektiv og forutsigbar drift

Jeg jobber med å automatisere gjentakende oppgaver for å redusere manuell innsats og sikre mer stabile og forutsigbare løsninger over tid.

Automatisering gjør det enklere å standardisere oppsett, redusere feil og få bedre kontroll på endringer i miljøet.

Jeg bruker scripting som et praktisk verktøy i drift, blant annet for å effektivisere oppgaver knyttet til brukere, servere og vedlikehold.

Eksempel: Automatisere opprettelse av brukere, konfigurasjon av servere og vedlikeholdsoppgaver, slik at drift blir mer robust og mindre avhengig av manuelle steg.

Azure DevOps

Gode pipelines gir tryggere leveranser, bedre kvalitet og raskere utvikling.

Fra kode til produksjon

Azure DevOps støtter hele leveranseprosessen, fra planlegging og versjonskontroll til bygg, deploy og forvaltning. Det gir bedre flyt og mer forutsigbare leveranser.

Jeg bruker pipelines til å automatisere både infrastruktur og applikasjoner, slik at endringer kan valideres og rulles ut på en kontrollert måte.

Eksempel: En pipeline kan først validere Terraform-kode, deretter kjøre plan, få godkjenning og så deploye endringer til test eller produksjon med full sporbarhet.

Containerteknologi

Containerteknologi gir fleksible og moderne måter å utvikle, deploye og drifte applikasjoner på.

Docker og Kubernetes

Jeg har grunnleggende erfaring med containerteknologi gjennom bruk av Docker og Kubernetes, og har også gjennomført kurs innen området.

Jeg har jobbet med å forstå hvordan applikasjoner kan pakkes i containere, samt hvordan Kubernetes brukes til å orkestrere og drifte slike løsninger.

Jeg har også brukt kubectl til å inspisere og håndtere ressurser i Kubernetes-miljøer.

For meg er containerteknologi et naturlig supplement til Azure, DevOps og automatisering, og noe jeg ønsker å bruke mer aktivt i fremtidige løsninger.

Eksempel: Docker kan brukes til å pakke applikasjoner med alle avhengigheter, mens Kubernetes håndterer deployering og skalering. Med kubectl kan man følge med på status og gjøre enkle operasjoner i miljøet.

Governance

Governance gir kontroll på kostnader, sikkerhet, ansvar og etterlevelse.

Kontroll uten å stoppe fremdrift

Governance handler om å finne balansen mellom styring og fleksibilitet. Med tydelige prinsipper, policyer og ansvar blir det enklere å bygge sikkert og effektivt.

Jeg jobber med strukturer som støtter standardisering, kostnadskontroll, sikkerhet og tydelig rollefordeling mellom teamene.

Eksempel: Policyer kan brukes til å kreve tagging, begrense hvor ressurser kan opprettes og sikre at kun godkjente tjenester og regioner brukes.

Sikkerhet

Sikkerhet må være innebygd i plattformen og støtte både drift og utvikling.

Zero Trust og innebygd sikkerhet

Jeg jobber med sikkerhetsprinsipper som Zero Trust, minst mulige privilegier, segmentering, identitetsstyring og sikker automatisering.

Jeg kombinerer Azure-sikkerhet med en strukturert tilnærming til risikoreduksjon, blant annet med utgangspunkt i NSMs grunnprinsipper for IKT-sikkerhet.

Målet er å redusere risiko, øke synlighet og samtidig gjøre det enklere å levere trygge løsninger i praksis.

Eksempel: Bruk av Entra ID, Conditional Access, Key Vault, logging, monitorering og policyer gir bedre kontroll på tilgang, hemmeligheter og hendelser i plattformen.

Hybrid drift

Mange virksomheter trenger løsninger som binder sammen on-prem og sky på en trygg måte.

Sky og eksisterende miljøer i samspill

Hybrid drift handler om å få identitet, nettverk, sikkerhet og tjenester til å fungere godt sammen på tvers av lokale miljøer og Azure.

Jeg har erfaring med å bygge bro mellom eksisterende infrastruktur og moderne skyplattformer, slik at virksomheten kan modernisere i riktig tempo.

Eksempel: Integrasjon mellom on-prem Active Directory og Entra ID, kombinert med sikre forbindelser til Azure, gir brukerne en mer sømløs og trygg hverdag.

CAF

Cloud Adoption Framework gir struktur for hele skyreisen, fra strategi til drift.

Cloud Adoption Framework

CAF hjelper virksomheter med å jobbe helhetlig med strategi, organisering, styring, sikkerhet, plattform og migrering. Det gir en tydelig retning for hvordan man lykkes i skyen over tid.

Jeg bruker CAF som et praktisk rammeverk for å sikre at tekniske valg henger sammen med virksomhetens behov og mål.

Eksempel: CAF kan brukes som grunnlag når en virksomhet skal planlegge landing zones, definere roller, etablere governance og prioritere hvilke tjenester som skal moderniseres først.

WAF

Well-Architected Framework bidrar til robuste, sikre og driftbare løsninger i Azure.

Well-Architected Framework

WAF brukes for å evaluere og forbedre arkitekturvalg. Det hjelper med å balansere krav til pålitelighet, sikkerhet, kostnadsstyring, drift og ytelse.

Jeg bruker Well-Architected Framework til å kvalitetssikre løsninger og til å finne forbedringsområder før små valg blir store problemer.

Eksempel: En løsning kan vurderes opp mot WAF for å se om den har tilstrekkelig redundans, riktig monitorering, kostnadskontroll og god nok sikkerhet før den tas i bruk i produksjon.
